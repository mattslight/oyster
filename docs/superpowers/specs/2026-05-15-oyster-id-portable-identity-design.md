# `.oyster/id` — portable source identity

**Status:** design approved, ready for plan
**Date:** 2026-05-15
**Author:** brainstorming session (Matthew + Claude)

## Context

PR [#490](https://github.com/oyster-to/oyster/pull/490) introduces the *"cwd is evidence; user organisation is authority"* model: longest-prefix session→source binding, `auto`/`manual` assignment modes, normalised source paths, "Path missing" chips, `set_source_path` MCP/REST, source consolidation on path collision. With #490, Oyster handles every local-path mutation case — renames, moves, missing paths, duplicate attaches — without conflating cwd with identity.

What #490 does **not** solve is the question *"is this folder on Machine A the same logical project as this folder on Machine B?"* Today the local row identity (`sources.id`) is a per-device UUID with no cross-machine meaning. After `git clone` on a second device, Oyster has no way to recognise the folder as a known project. This blocks any future source-metadata cross-device sync ([#296](https://github.com/oyster-to/oyster/issues/296)) and any UI grouping that wants to collapse worktrees / sibling checkouts under one logical project heading.

`.oyster/id` is the primitive that establishes portable identity for a source folder. It is **disk-backed**, **opt-in to travel via git**, and **deliberately separate from local row identity**. v1 ships the schema, the read/write helpers, the integration into `addSource` / `scanSpace`, and a one-time migration for existing rows. **No UI surface** is introduced in v1.

## Goals

- Give every attached source folder a portable UUID that lives on disk at `<source-path>/.oyster/id`.
- Persist that UUID in the DB as `sources.portable_id`, separate from `sources.id`.
- Allow multiple local source rows to share the same `portable_id` — this is the **intended state** for worktrees and sibling checkouts.
- Establish the primitive cleanly enough that future cross-device source-metadata sync ([#296](https://github.com/oyster-to/oyster/issues/296)) can use `portable_id` as a grouping/reconciliation key.

## Non-goals (v1)

- **No UI surface.** No badges, chips, inspector rows, or grouping displays. v1 is schema + file + helpers only.
- **No auto-commit.** Oyster never `git add`s, `git commit`s, or edits `.gitignore` on the user's behalf. The user discovers `.oyster/id` via `git status` and decides what to do with it.
- **No cross-device sync.** That's [#296](https://github.com/oyster-to/oyster/issues/296). This spec just lays the groundwork.
- **No re-solving local path issues.** #490 owns rename / move / missing-path / consolidate flows; this spec doesn't touch them.
- **No session/artefact rebinding.** `sources.id` remains the only FK target for `sessions.source_id` and `artifacts.source_id`. `portable_id` is never referenced by other tables.

## Core invariants (must hold in every code path)

1. **`portable_id` is never unique.** Not locally, not in any future cloud-sync schema. Multiple `sources` rows can — and routinely will — share one `portable_id` (worktrees, duplicate checkouts, sibling clones on the same machine).
2. **Future sync must not use `(account_id, portable_id)` as a unique key.** Use it as a grouping/reconciliation key only. Each device's local source remains its own instance.
3. **Writing `.oyster/id` may dirty a git repo.** Oyster creates the file silently as part of the attach flow but **never** auto-commits, **never** edits `.gitignore`, and **never** asserts cross-device identity works until the user commits the file themselves. The user owns the git decision.
4. **Portable identity is disk-backed, not imaginary.** If Oyster cannot write `.oyster/id` (read-only FS, perms, etc.), `sources.portable_id` stays `NULL`. We do not invent a DB-only id; the disk is the source of truth.
5. **`sources.id` is never derived from anything external and never mutated.** It is generated by `crypto.randomUUID()` at `addSource` time and stays for the life of the row.
6. **`.oyster/` is internal metadata, never user-visible content.** The scanner must skip it explicitly. Its contents must never appear as an artefact tile, never be searchable, never be published, and never surface in any user-facing list of files. Today's dot-prefix exclusion is implicit; this spec makes it explicit and defensive.

## Architecture

### The two identities

| Column | Purpose | Uniqueness | Mutability | Sourced from |
|---|---|---|---|---|
| `sources.id` | Local row primary key. FK target for `sessions.source_id`, `artifacts.source_id`, and #490's binding logic. | Unique per row, per device. | Never mutated. | `crypto.randomUUID()` on attach. |
| `sources.path` | Specific local checkout / worktree / folder path. | Soft-unique (one active row per path; #490 owns collision handling). | Mutable via #490's `set_source_path`. | User. |
| **`sources.portable_id`** (new) | Cross-machine identifier — *"these folders represent the same logical project."* | **Not unique.** Multiple rows can share it. | Updated in-place when `.oyster/id` changes (single-row UPDATE, no cascade). | `<path>/.oyster/id`. |

### On disk

```
<source-path>/
  .oyster/
    id              ← plain UTF-8, one canonical UUID v4 (lowercase, hyphenated), trailing newline
```

That is the entire on-disk surface for v1. No `.oyster/config`, no `.oyster/local/`, no `.oyster/.gitignore`. Those are future markers; YAGNI for v1.

### Schema migration

```sql
ALTER TABLE sources ADD COLUMN portable_id TEXT NULL;
CREATE INDEX IF NOT EXISTS sources_portable_id
  ON sources(portable_id)
  WHERE portable_id IS NOT NULL;
```

Index is partial because `NULL` rows (sources where the write failed) don't need to participate in grouping lookups.

### Adoption rule — file wins, no cascade

On any path that reads `.oyster/id` (attach or scan), if the file's UUID differs from `sources.portable_id` for the matching local row, the resolution is a **single-row UPDATE** on `sources`:

```sql
UPDATE sources SET portable_id = <file_id> WHERE id = <local_row_id>;
```

That is the entire reconciliation. **No cascade.** `sessions.source_id` and `artifacts.source_id` are unaffected because they FK to `sources.id`, which doesn't change. **No UX surface** — the swap is silent because nothing user-observable changes (paths stay the same, labels stay the same, only an internal grouping key updates).

### Worktree behaviour (intentional)

```
~/Dev/oyster-dev/.oyster/id                          = abc-...
~/Dev/oyster.worktrees/feature-490/.oyster/id        = abc-...
```

Both are valid `sources` rows. Their `sources.id` values differ (`local-A`, `local-B`). Their `sources.portable_id` values are the same (`abc-...`). Sessions and artefacts bind to the per-row `sources.id`. A future grouping UI may collapse them under one project heading; today the rows render as independent sources.

### Scanner exclusion (invariant 6)

`server/src/space-service.ts` already maintains `SKIP_DIRS = Set(["node_modules", ".git", "dist", ...])` and the walker has a belt-and-braces check at the per-entry level (`if (SKIP_DIRS.has(entry) || entry.startsWith(".")) continue`). This spec adds `".oyster"` to `SKIP_DIRS` explicitly — even though the dot-prefix rule already excludes it implicitly, the explicit entry is the defensive guarantee that surviving a future change to dot-handling won't accidentally expose `.oyster/id` as an artefact.

Consequence: `.oyster/id` never reaches `upsertCandidate`, never gets an artefact row, never appears in tiles / search / publish / inspector listings. It exists on disk and in `sources.portable_id`, and nowhere else.

### Interaction with #490 (merged)

- **Path mutation** (`set_source_path`): unchanged. Renaming or moving a folder doesn't touch `portable_id` — `.oyster/id` lives inside the folder bytes and moves with them.
- **Path existence is advisory** (#490's relaxed precondition on `addSource`): a folder attached before it exists on disk (orphan-tile attach for a renamed-on-disk folder) inserts with `portable_id = NULL`. We can't read or write `.oyster/id` for a folder that isn't there. After the user runs "Update folder location…" and the next scan finds the real folder, `portable_id` populates from the file.
- **Path-missing chip**: unchanged surface; independent of `portable_id`.
- **Longest-prefix session binding**: unchanged. Sessions bind to local `sources.id` via cwd. `portable_id` is never consulted by binding logic.
- **Source consolidation** (`consolidate_sources`): operates on local `sources.id` and is independent of `portable_id`. Two rows with the same `portable_id` are *not* consolidation candidates — they're the intended worktree state.

## Components

### New module — `server/src/oyster-id.ts` (~80 lines)

```typescript
const OYSTER_DIR = ".oyster";
const ID_FILE = "id";

// Discriminated result so callers can branch on cause, not collapse every
// non-happy-path into `null`. The error-handling table in this spec
// requires per-cause behaviour (overwrite vs preserve vs leave-unchanged),
// which a `string | null` return would erase.
type OysterIdReadResult =
  | { status: "valid"; id: string }
  | { status: "missing" }
  | { status: "malformed"; value?: string }
  | { status: "unreadable"; error: unknown }
  | { status: "blocked"; reason: ".oyster-is-file" };

// Read <root>/.oyster/id and classify. Never throws.
//   - valid      → adopt the id
//   - missing    → caller may generate + write
//   - malformed  → caller logs warning, must NOT overwrite (preserve state)
//   - unreadable → caller leaves portable_id NULL / unchanged
//   - blocked    → caller leaves portable_id NULL (`.oyster` exists as a file)
readOysterId(root: string): OysterIdReadResult

// Atomic write via tmpfile + rename. Creates .oyster/ if needed.
// Throws on filesystem error so callers see the failure explicitly and
// can decide what to record on the row (per invariant 4: disk is truth).
writeOysterId(root: string, id: string): void

// True if `s` matches the canonical 8-4-4-4-12 UUID v4 shape.
isValidUuid(s: string): boolean
```

No state, no class, no side-effects beyond the named filesystem write.

### Integration — `server/src/space-service.ts`

**`addSource(spaceId, rawPath)`** — decide `portable_id` *before* the INSERT, never after. The row is written exactly once; we don't INSERT a generated id and then UPDATE it back to `NULL` if the disk write fails. This preserves invariant 4 (disk is truth).

The control flow:

1. `resolved = normaliseSourcePath(rawPath)` (existing, unchanged from #490).
2. **If `resolved` does not exist on disk** (#490's advisory-existence case):
   - `portable_id = NULL`. No `readOysterId`, no `writeOysterId` — there's no folder to read from or write to.
   - INSERT row. Done. A later scan after the user runs "Update folder location…" will populate `portable_id` from disk.
3. **Otherwise**, branch on `readOysterId(resolved)`:
   - `{ status: "valid", id }` → `portable_id = id`. No write needed.
   - `{ status: "missing" }` → generate `newId = crypto.randomUUID()`. **Attempt** `writeOysterId(resolved, newId)`.
     - Write succeeds → `portable_id = newId`.
     - Write fails → log warning, `portable_id = NULL` (we do not store an id that isn't on disk).
   - `{ status: "malformed" }` → log warning, `portable_id = NULL`. Never overwrite the malformed file.
   - `{ status: "unreadable" }` → log warning, `portable_id = NULL`. The file might exist but we can't see it; assume the user owns whatever's there.
   - `{ status: "blocked" }` (`.oyster` is a regular file) → log warning, `portable_id = NULL`. Treat as user state we don't understand.
4. `sources.id = crypto.randomUUID()` (always — never derived from anything external).
5. **Single INSERT** with the decided `(id, path, space_id, portable_id, ...)`. Never followed by a corrective UPDATE.

If a sources row with the same `portable_id` already exists on this device — fine. No special handling. They're worktrees / sibling checkouts.

**`scanSpace(spaceId)`** — once per source at the top of the per-source loop, before `walk()`:

1. `result = readOysterId(source.path)`
2. Branch on the status:
   - `valid` and `result.id ≠ source.portable_id` → single-row UPDATE, set `portable_id = result.id`.
   - `valid` and `result.id == source.portable_id` → no-op.
   - `missing` and `source.portable_id IS NULL` → attempt `writeOysterId` with a fresh UUID; on success, single-row UPDATE.
   - `missing` and `source.portable_id` is already set → leave the DB alone (file was probably deleted manually; the user's intent isn't clear; don't clobber what's in the DB).
   - `malformed` → log warning, leave `source.portable_id` unchanged. Never overwrite.
   - `unreadable` → log warning, leave `source.portable_id` unchanged. Transient perm issues mustn't clobber known state.
   - `blocked` → log warning, leave `source.portable_id` unchanged.
3. Continue with the scan.

### Migration — `server/src/db.ts` (or `server/src/migrations/oyster-id.ts`)

Idempotent, runs once per boot after schema migrations:

```typescript
for each row in sources where portable_id IS NULL:
  if !existsSync(row.path):
    continue                              // can't write; #490 will surface "Path missing"
  switch (readOysterId(row.path)) {
    case "valid":
      UPDATE sources SET portable_id = result.id WHERE id = row.id
      break
    case "missing":
      newId = crypto.randomUUID()
      try {
        writeOysterId(row.path, newId)
        UPDATE sources SET portable_id = newId WHERE id = row.id
      } catch fs error {
        log warning; leave portable_id NULL
      }
      break
    case "malformed":
    case "unreadable":
    case "blocked":
      log warning; leave portable_id NULL    // never overwrite; never invent
      break
  }
```

Migration **never** mutates `sources.id`. It only fills in `portable_id` for rows where it's `NULL` AND the disk write succeeds. Safe to re-run; safe to interrupt.

## Data flow

### Attach (new source, folder exists, no `.oyster/id` yet)
```
addSource → path exists; readOysterId() → { status: "missing" }
         → generate portable UUID
         → writeOysterId(path, newId)
         →   success → portable_id = newId
         →   failure → portable_id = NULL (no row UPDATE — INSERT once)
         → INSERT sources row exactly once
```

### Attach (folder already has `.oyster/id`, e.g. fresh `git clone`)
```
addSource → path exists; readOysterId() → { status: "valid", id: "abc" }
         → portable_id = "abc"; no file write needed
         → INSERT sources row exactly once
```

### Attach (folder does NOT exist on disk — orphan-tile / renamed-on-disk, per #490)
```
addSource → !existsSync(path) → portable_id = NULL
         → no file read, no file write
         → INSERT sources row exactly once
         → (later) user runs "Update folder location…", scan picks up .oyster/id, UPDATE fills in portable_id
```

### Scan with file updated by external means (e.g. git pull brought a new id)
```
scanSpace → for each source:
         →   readOysterId(source.path) → "xyz"
         →   source.portable_id was "abc"
         →   UPDATE sources SET portable_id="xyz" WHERE id=source.id
         →   sessions and artefacts unchanged (still FK to source.id)
```

### Migration on first boot post-upgrade
```
boot → for each source where portable_id IS NULL:
         if !existsSync(path)                    → skip
         else switch on readOysterId(path):
           valid       → UPDATE row.portable_id = result.id
           missing     → generate UUID; if write succeeds, UPDATE; otherwise skip
           malformed   → log; skip (never overwrite)
           unreadable  → log; skip
           blocked     → log; skip
```

### Detach
Unchanged from today. `removeSource` soft-deletes the row. `.oyster/id` on disk is **left in place** — Oyster doesn't clean up the user's filesystem on detach.

## Error handling

Indexed by `readOysterId` result status (plus the path-missing branch that bypasses the read entirely).

| Condition | `addSource` | `scanSpace` | Why |
|---|---|---|---|
| Path doesn't exist on disk (#490 advisory case) | INSERT with `portable_id = NULL`. No read, no write. | n/a — row won't be reached for scan until path exists. | We don't fabricate identity for folders that aren't there. |
| `{ status: "valid", id }` | Adopt: `portable_id = id`. No file write. | If differs from `source.portable_id`, single-row UPDATE. If matches, no-op. | File is truth. |
| `{ status: "missing" }` | Generate UUID, attempt write. **Only set `portable_id` if the write succeeds**; otherwise NULL. | If `source.portable_id` is NULL, attempt generate + write; otherwise leave the DB alone (don't clobber on transient delete). | Per invariant 4: disk-backed, never imaginary. |
| `{ status: "malformed", value }` | Log warning. `portable_id = NULL`. **Never overwrite** the malformed file. | Log warning. Leave `source.portable_id` unchanged. | Don't silently destroy user state we don't understand. |
| `{ status: "unreadable", error }` | Log warning. `portable_id = NULL`. | Log warning. Leave `source.portable_id` unchanged. | Transient perm issues mustn't clobber known state or invent absent state. |
| `{ status: "blocked" }` (`.oyster` exists as a file, not a directory) | Log warning. `portable_id = NULL`. No cleanup. | Log warning. Leave `source.portable_id` unchanged. | Defensive: user might have something at that name we don't recognise. |
| `writeOysterId` throws (read-only FS, no perms, disk full) | `portable_id = NULL`. Source still attached and usable. | Same. | Per invariant 4. |
| Same `portable_id` already in DB on another local row | No-op. This is the worktree / sibling-checkout case. | Same. | Per invariant 1. |

## Future sync implications (preview — not in this spec's scope)

When [#296](https://github.com/oyster-to/oyster/issues/296) builds source-metadata cross-device sync, `portable_id` is the join key:

- Mac: `sources(id=local-A, portable_id=abc, path=/Users/m/Dev/oyster-dev)`
- Windows: `sources(id=local-B, portable_id=abc, path=C:\Users\m\Dev\oyster-dev)`

These are recognised as the same logical project, but each device retains its own local row, its own path, and its own session/artefact bindings. The sync engine reconciles **metadata** (label, group, pin state, share-token) across rows sharing a `portable_id`; it does **not** merge rows, and it does **not** treat `(account_id, portable_id)` as a unique source instance (per invariant 2).

## Out of scope (deferred to later)

- **UI grouping.** Read-side display that collapses same-`portable_id` rows under a project heading. Trivial to add later off the existing column + index.
- **Storage-mode badge.** Separate feature ([`docs/mockups/storage-mode-badge.html`](../../mockups/storage-mode-badge.html)) that becomes load-bearing once cross-device sync is observable.
- **Move-to-Oyster-vault action.** Cut from the storage-mode badge brainstorm; no current use case until cross-device sync ships.
- **`.oyster/config`, `.oyster/local/`, etc.** Future markers. Not in v1.
- **Auto-commit / auto-`.gitignore`.** Forbidden by invariant 3.
- **MCP / REST surface for `portable_id`.** Not needed for v1; consumers should use `sources.id` as today.
- **Watcher-driven detection of `.oyster/id` changes.** v1 is re-scan-driven; if a `git pull` brings a new id while Oyster is running, the user sees it on the next scan.

## Test plan

Tests live under `server/test/`. The seven required scenarios:

1. **Attach with no `.oyster/id`.** Folder is empty. `addSource` succeeds; `sources.id` is a fresh UUID; `sources.portable_id` matches `<path>/.oyster/id` contents.
2. **Attach with existing `.oyster/id`.** Folder has `.oyster/id` containing valid UUID `X`. `addSource` succeeds; `sources.portable_id = X`; the file is **not** overwritten.
3. **Two sources with the same `.oyster/id` (worktree case).** Attach `~/repo` (writes id `abc`), then attach `~/repo-worktree` after copying `.oyster/id` to it. Both rows exist; both have `portable_id = abc`; sessions/artefacts bind correctly to each row's `sources.id`.
4. **Scan updates `portable_id` when the file changes.** Attach a folder (gets `portable_id = abc`). Externally rewrite `.oyster/id` to `xyz`. Call `scanSpace`. `sources.portable_id` updates to `xyz`. `sources.id` is unchanged. No `sessions` or `artifacts` rows are touched.
5. **Malformed `.oyster/id`.** Folder has `.oyster/id` containing `"not a uuid"`. `addSource` succeeds; `sources.portable_id = NULL`; warning logged; the file is **not** overwritten. Scan path: same — `portable_id` left unchanged when the file is malformed.
6. **Sessions and artefacts remain bound to `sources.id`** (invariant test). Insert a source + session + artefact. Trigger every code path that touches `portable_id` (attach, scan, migration, file rewrite). Assert `sessions.source_id` and `artifacts.source_id` are byte-identical to their initial values across all paths.
7. **Migration never mutates `sources.id`** (invariant test). Pre-seed `sources` with rows that have `portable_id IS NULL`. Run the migration. Assert: every row's `sources.id` is unchanged (compared against pre-migration snapshot); every row's `portable_id` is either a valid UUID (read from file or freshly written) or `NULL` (if write failed and we can simulate that).
8. **Attach to a non-existent path (#490 advisory case).** Call `addSource` with a `rawPath` that doesn't exist on disk. Assert: the row is inserted exactly once; `portable_id = NULL`; no filesystem read or write was attempted (verify by spying on `readOysterId` / `writeOysterId`). Then create the folder, run `scanSpace`, assert `portable_id` populates from the now-existing `.oyster/id` (or from a freshly written one if the folder didn't have it yet).
9. **`addSource` writes the row exactly once.** Cover the two write-failure branches (`status: "missing"` + write throws; path exists + perms simulate read-only). Assert the row is INSERTed once with `portable_id = NULL`, and there is **no** follow-up corrective UPDATE on `portable_id` from the addSource path.
10. **`.oyster/` is never registered as an artefact** (invariant 6). Call `addSource` on a folder containing only a few `.md` files. Confirm `.oyster/id` gets written. Run `scanSpace`. Assert: the resulting artefacts include the `.md` files but contain **zero** rows whose path resolves into `.oyster/` (no `.oyster`, no `.oyster/id`, no `.oyster/anything`). Repeat with a manually-pre-created `.oyster/id` to cover the adoption path. Also assert `.oyster` is present in `SKIP_DIRS` directly.

**Failure-mode coverage** (smaller tests, same surface):

- Read-only filesystem on attach → `portable_id = NULL`, source still attached, no orphan UPDATE.
- `.oyster/` is a regular file (`status: "blocked"`) → `portable_id = NULL`, log warning, no cleanup.
- Malformed `.oyster/id` content → `portable_id = NULL` on attach; on scan with an existing valid `portable_id`, the malformed read leaves the DB untouched.
- Unreadable file (chmod 000) → same as malformed for state outcomes.
- Concurrent `addSource` calls on the same path → standard `sources.path` uniqueness wins (no new race introduced by this work).
