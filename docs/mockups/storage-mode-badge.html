Storage-mode badge — three states, one action

v1 answers exactly one question: "where are my bytes?" It does not tell the user whether their work is safe. It does not detect git remotes. It does not warn about disk-failure risk. That entire mental model is v1.5 territory and is what was metastasising the spec.

Engineering

Detection — two checks

Two filesystem checks per attached source; result persisted on the sources row.

artefact ├─ source_id IS NULL ─► VAULT (shield) └─ source_id IS NOT NULL ├─ ancestor has .git/? │ ├─ yes ─► GIT FOLDER (git icon) │ └─ no ─► FOLDER (folder icon)
Scope

What ships

  1. Schema: add storage_mode column to sources. Enum: git | folder. Vault is implicit from artifacts.source_id IS NULL.
  2. Detection on attach: check for .git/ ancestor. One filesystem call.
  3. StorageIcon component (~20 lines): inline 14px icon + title tooltip. Reused in ProjectTile, ArtefactTable, and the inspector header.
  4. Inspector Storage + Path rows. Two new rows in the inspector body.
  5. Move to Oyster vault action: copies the file into ~/Oyster/spaces/<space>/<id>/, soft-deletes the source-linked artefact, creates a vault artefact pointing at the new path. One MCP-shaped call.

Explicitly out of v1: git remote detection · pushed/unpushed indicators · backup judgement · "at risk" copy · Initialize-git action · "no off-machine backup" warnings · disk-failure framing. All revisit at v1.5 once we've seen how people actually use the legibility version.
