The workspace-for-agents category map

Twenty-plus products are now circling the same idea: a place that sits above your AI agents — sees what they do, remembers what they wrote, organises what they made. Anthropic shipped half of it inside Claude Code itself last week. A 75k-star indie project pitches almost exactly what Oyster pitches. Two other repos already use the word “OS” in their names. This is the May 2026 picture, with Oyster placed honestly in it.

TL;DR

There is no single product that does what Oyster aspires to do — “every agent, one shared brain, on every device, with a workspace and publish/share.” But there are now twenty-plus products doing one or two of the pieces, several with massive star counts, and one (thedotmack/claude-mem, 75.6k stars) pitching literally Oyster's tagline.

The category splits roughly into four overlapping subsets — multi-session managers, memory layers, workspace / chat platforms, and agent OSes / orchestrators. Most products sit cleanly in one. A few sit in two. None sit in all four. That “all four” is the umbrella position Oyster is claiming, and it is, today, defensible — but only if Oyster (a) closes the multi-agent-ingestion gap that CCManager and Claude Squad already have, (b) doesn't get drowned by name collision with two other “OS”-suffixed products (OpenFang 17.5k, brobertsaz/claude-os 287), and (c) ships the cross-device + memory + publish trio before anyone else thinks to bundle them.

The biggest single risk: claude-mem at 75.6k stars, pitching multi-agent memory injection with the same “works with Claude Code, OpenClaw, Codex, Gemini, Hermes, Copilot, OpenCode + More” multi-agent claim. Real verification of the codebase shows it is currently Claude-Code-first (the README leads with “built for Claude Code”) — but the marketing pitch and 75k-star scale mean Oyster is racing a much larger boat to the same harbour. The defensible reply isn't “we also have memory” — it's workspace + spaces + artefacts + publish + cross-device sync, none of which claude-mem ships. Add Cursor / Codex / OpenCode watchers fast.

The biggest single opportunity: the umbrella is empty. Twenty competitors each own one slice. Oyster can own the union — but only if it commits to it visibly and stops shipping the Sprint-2 product underneath the Sprint-5 box (see §7).

01What category are we even in?

Three overlapping observations, each one a category-defining design choice:

  1. The chat-log-as-UI is broken. If you run more than one Claude Code session, or use Claude Code and Cursor and Codex, you can't see what they're all doing without flipping between terminals. Products in this subset solve “one screen for many sessions”: claude agents (Anthropic, native), Claude Squad, CCManager, Vibe Kanban, agent-deck, Conductor, OpenClaw OS, the various dashboards. This is a commoditising category.
  2. Models forget between sessions. Anything you tell the model on Tuesday is gone Wednesday morning. Products in this subset solve “persistent memory across conversations”: claude-mem, brobertsaz/claude-os, mem0, Letta, Zep/Graphiti, Cognee. This subset has both consumer products and infrastructure libraries — they don't compete head-on with each other.
  3. Work needs a place to live. Artefacts the agent produced — notes, dashboards, decks, diagrams — need somewhere durable to sit between conversations. Products in this subset are the workspace itself: Open WebUI, LobeChat, AnythingLLM (with workspaces), LibreChat (with artifacts), OpenClaw OS, vibeyard. This subset is large, mature, and mostly chat-shaped.

A fourth, smaller subset is emerging:

  1. Agent orchestration as “an OS”. A few products explicitly position themselves as an “Agent OS” — OpenFang (17.5k stars), OpenAgents (3.5k), OpenClaw OS (0.1k), brobertsaz/claude-os (0.3k). None of them mean the same thing by “OS” — three different products have decided that label fits — but the word is conspicuously crowded. This matters for Oyster's brand.
Oyster's design claim is the union of all four subsets. “Every agent, one shared brain” spans (1) multi-session and (2) memory. Spaces + artefacts + publish span (3) workspace. The free + Pro tier spine and cross-device sync structurally aim at (4) being “the OS” in the everyday sense — the thing that sits across all your machines and remembers everything. That union is empty today. It is also where you get burnt fastest if you don't pick which slice to ship best first.

03Profiles, grouped by subset

3.1 — Multi-session / agent managers (closest neighbours)
Anthropic claude agents (agent view) Built into Claude Code 2.1.139+ · research preview · May 2026 High threat

“One screen for every Claude Code session you have running.”

Native to Claude Code itself. Supervisor process keeps background sessions alive across terminal close. Per-session worktrees automatic, PR status dots, peek-and-reply, dispatch via claude --bg or /bg. Terminal-only. “Sessions are local” explicitly — no cross-device.

What it kills for Oyster: the “watch your Claude Code sessions in a browser” framing as a headline. Anyone happy in a terminal and only using Claude Code is now sufficiently served.

What it doesn't kill: cross-agent (Anthropic owns Claude only, structurally), memory across sessions, cross-device, browser UI, spaces, artefacts, publish — none of these are in claude agents.

claude-mem (thedotmack/claude-mem) 75,602 stars · Apache-2.0 · created 2025-08-31 · v6.5.0 · pushed 2026-05-13 High threat

“Persistent Context Across Sessions for Every Agent — Captures everything your agent does during sessions, compresses it with AI, and injects relevant context back into future sessions. Works with Claude Code, OpenClaw, Codex, Gemini, Hermes, Copilot, OpenCode + More.”

This is the closest single product to Oyster's pitch. 75.6k stars in 9 months. Apache-2.0. Active (pushed yesterday). The GitHub-description pitch is exactly Oyster's: every agent, persistent context, cross-session. README leads with “Persistent memory compression system built for Claude Code” — so the multi-agent claim is partly aspirational in the actual code, but the marketing surface and 75k-star scale mean Oyster is racing a much bigger boat to the same harbour. Auto-compresses session content with an LLM, stores in SQLite + Chroma, injects via Claude Code session-start hook.

What it doesn't ship: workspace UI, spaces, artefacts, publish, cross-device sync, generative UI. It is purely a memory plumbing layer. Oyster's defensible reply isn't “we have memory too” — it's “our memory lives inside a workspace, with spaces, artefacts, publish, and sync.”

Brand-name note: “built for Claude Code” in the README + 75k stars means the term “Claude memory” is going to keep pointing at this product, not Oyster, for the foreseeable future.

Claude Squad (smtg-ai/claude-squad) 7,453 stars · AGPL-3.0 · created 2025-03 · pushed 2026-03-28 Medium threat

“Manage multiple AI terminal agents like Claude Code, Codex, OpenCode, and Amp.”

Terminal app on top of tmux + git worktrees. Same scope as claude agents but multi-agent (Claude Code, Codex, Gemini, Aider). Yolo / auto-accept mode. Each task gets isolated worktree. Same AGPL licence as Oyster. This is Oyster's roadmap #298 already shipping, in a terminal.

What it doesn't ship: memory, workspace, spaces, artefacts, publish, cross-device, browser UI. The terminal-only constraint is real — Claude Squad's audience is the dotfile-pinning power-user crowd. Oyster competes for the browser-native audience.

CCManager (kbwo/ccmanager) 1,099 stars · MIT · created 2025-06 · pushed 2026-05-13 Medium threat

“Coding Agent Session Manager for Claude Code / Gemini CLI / Codex CLI / Cursor Agent / Copilot CLI / Cline CLI / OpenCode / Kimi CLI.”

Eight agents in one terminal. No tmux dependency (vs Claude Squad). Real-time session status (busy / waiting / idle). Worktree-based isolation. Multi-project. Auto-approve experimental. This is what Oyster's 0.9.0 multi-agent ingestion has to compete with — and CCManager has it shipping today.

What it doesn't ship: browser UI, memory, artefacts, publish, cross-device, spaces above agent. Still terminal-only. But the “every agent” claim — which Oyster wants to be famous for — already has a working terminal-native product behind it.

Vibe Kanban (BloopAI/vibe-kanban) 26,213 stars · Apache-2.0 · company shut down · OSS continues Context (cautionary)

“Get 10X more out of Claude Code, Codex or any coding agent.”

Kanban board for AI agents. Each agent works in its own workspace (branch + terminal + dev server). Cloud + remote access. Bloop (the company) shut down in April 2026. Per their farewell post: “thousands of software engineers use Vibe Kanban every day to ship more with coding agents, but the vast majority are free users and the company couldn't find a business model.” The project survives as community-maintained OSS; cloud features sunset after 30 days, transitioning to fully local.

The strategic lesson for Oyster: 26k stars and thousands of daily-active users wasn't enough to monetise. This is the most direct data point in the category about “is this monetisable?” and the answer was no for Bloop. Worth understanding why: free users couldn't be converted, the value was thin on top of free coding-agent CLIs. Oyster's Pro tier (cross-device sync, durability, publish caps) targets a different paywall — “your data follows you,” not “your agents are organised.” That's a stronger paywall, but only if R1/R3/R4/R7 are actually shipped and verifiable.

agent-deck (asheshgoplani/agent-deck) 2,401 stars · MIT · created 2025-12 · pushed 2026-05-14 Low threat

One TUI for Claude, Gemini, OpenCode, Codex, and more. Same shape as Claude Squad / CCManager; smaller, MIT, newer.

OpenClaw OS (thesysdev/openclaw-os) 119 stars · MIT · created 2026-03-27 Low threat (different scope)

Workspace plugin for OpenClaw — the 371k-star personal AI assistant. Two-surface generative-UI runtime: inline UI (openui-lang in chat replies) and Apps (durable, reactive, with Query/Mutation/$state, exec/read/SQLite from the runtime, cron-driven data scripts, version history, restore). Built on Thesys's MIT-licensed OpenUI (5.4k stars), which Oyster could embed.

What's distinctive: the strongest generative-UI primitive in the open-source AI tooling ecosystem. PWA + push. Lint loop + skill-gate enforcing the DSL.

Strategic relevance to Oyster: not a direct competitor (different user base, lives inside OpenClaw). The relevant move is to borrow openui-lang as a future-state artefact renderer — see §7.

Smaller dashboards & tools Various · mostly <1k stars Low individually

Conductor (rmindgh, 4★), Claudia (marcusbey, 27★, GUI app), claude-control (sverrirsig, 116★, macOS), seunggabi/claude-dashboard (30★, k9s-style TUI), Stargx/claude-code-dashboard (9★), Tpain166/claude-dashboard (2★), amahpour/claude-code-command-center (4★), vibeyard (1k★, “IDE built for AI coding agents”).

Each is small alone. Collectively they signal the category is saturating fast. The pattern is consistent: every two weeks another developer ships a Claude Code dashboard. The mid-tier of this list (1k–10k stars) is the “watching Claude Code sessions” commodity layer Anthropic has now started absorbing natively.

3.2 — Memory layers

Split into consumer products (drop-in for Claude Code) and infrastructure libraries (build memory into your own agent).

brobertsaz/claude-os 287 stars · MIT · created 2025-10 · v2.5 · live at thebob.dev/claude-os Medium threat (name + memory)

“Give Your AI a Memory — Claude Code that actually remembers you.”

Python + TypeScript. Natural-language memory (“remember this…”), cross-KB search via MCP, tree-sitter indexing (10k files in 30s), 36+ community skills via Smithery, knowledge lifecycle (dedupe / consolidate / archive / health), real-time Kanban board, jobs dashboard, 100% local with Ollama.

Why it matters: the literal brand-name collision. Two products called “Claude OS” / “Oyster OS” in the same adjacent space, neither bare-noun. Google for “Claude OS” today lands here. Drop the “OS” in Oyster's brand — see §7.

mem0 (mem0ai/mem0) 55,658 stars · Apache-2.0 · 2023 vintage Context (infrastructure)

“Universal memory layer for AI Agents.”

Provider-agnostic memory library + cloud SaaS. Four-scope memory model (user_id / agent_id / run_id / app_id). MCP server for Claude Code integration. Anyone building memory in their AI app probably uses mem0 (or Letta, Zep, Cognee). Oyster's home-grown FTS5 memory store competes with this at the infrastructure level — and the gap will widen as mem0 keeps shipping benchmarks and personalisation features.

Strategic implication: Oyster's R2 semantic recall (0.9.0 roadmap) needs to either (a) embed mem0 / Letta / Cognee under the hood, or (b) ship a credible home-grown alternative. The R&D cost of (b) is high; the integration cost of (a) is low. The roadmap doc doesn't yet name which.

Letta (formerly MemGPT, letta-ai/letta) 22,704 stars · Apache-2.0 · 2023 vintage Context (infrastructure)

“Platform for stateful agents.” Agent runtime built around tiered memory — agents actively manage their own memory, decide what to archive, rewrite their system prompts based on past interactions. Strongest for long-horizon agents that run for days.

Zep / Graphiti (getzep/zep, getzep/graphiti) 4,566 / 26,042 stars · Apache-2.0 Context (infrastructure)

Real-time temporal knowledge graphs for AI agents. Stores facts with timestamps and relationships — “I used to live in London, moved to Tokyo” is understood as a state change. Strongest for memory with temporal queries.

Cognee (topoteretes/cognee) 17,218 stars · Apache-2.0 Context (infrastructure)

“Memory control plane for AI Agents in 6 lines of code.” Graph-based cross-session persistence. Strongest for deep knowledge retrieval and unstructured-document ingestion.

3.3 — Workspace / chat platforms (the surface)
Open WebUI (open-webui/open-webui) 136,994 stars · 2023 vintage · self-hosted Context (different shape)

The dominant self-hosted chat UI. Chat-first. Multi-provider. MCP support. Not a workspace in Oyster's sense — it doesn't have spaces, artefacts as first-class, or external-agent watching. But it owns the “self-hosted Claude UI” mindshare and would be the first thing many users would compare Oyster to if Oyster's chat bar stays prominent.

LobeChat (lobehub/lobe-chat) 77,046 stars · 2023 vintage Context (different shape)

“Ultimate space for work and life — find, build, collaborate with agents.” Most aggressive “agentic workspace” pitch among the chat platforms. PWA. Plugin marketplace. Still chat-shape underneath.

AnythingLLM (Mintplex-Labs/anything-llm) 60,016 stars · MIT Low–medium threat

Owns the term “workspace” in the self-hosted AI chat world. Workspaces are isolated environments each with their own documents, model config, and conversation history. The closest to Oyster's “Spaces” concept. Documents-plus-chat is the primary interaction; the agent doesn't watch anything external, doesn't publish, doesn't sync across devices.

Implication for Oyster: the word “workspaces” means “chat folders with documents” to 60k people. Oyster's Spaces should be positioned distinctly — they're projects, with attached repos, attributed sessions, watched agents, and synced state. Not chat folders.

LibreChat (danny-avila/LibreChat) 36,976 stars · MIT Low threat

Multi-provider chat. Strong MCP integration. Has “Code Artifacts” — renders React, raw HTML, Mermaid in browser. That's the closest analogue in the chat-platform world to OpenClaw OS's inline-UI surface and Oyster's HTML artefact tiles. The pattern is converging.

3.4 — Agent OS / orchestration
OpenFang (RightNow-AI/openfang) 17,508 stars · Apache-2.0 · created 2026-02-24 Medium threat (brand-name)

“Open-source Agent Operating System.”

17.5k stars in three months. Self-describes as an “Agent OS.” Three native drivers (Anthropic, Gemini, OpenAI-compatible) routing to 27 providers, with intelligent routing including task complexity scoring, automatic fallback, and cost tracking. This is the third product using “OS” in the same neighbourhood (Oyster OS, brobertsaz/claude-os, OpenFang). The term is now thoroughly diluted.

Implication for Oyster: the longer Oyster keeps “OS” in the brand, the more it gets averaged-out with OpenFang's routing-layer + brobertsaz's memory-tool. None of them mean the same thing by “OS” — which is exactly the problem.

OpenAgents (openagents-org/openagents) 3,462 stars · Apache-2.0 Low threat

“AI Agent Networks for Open Collaboration.” Interactive terminal dashboard (agn) for installing runtimes, configuring API keys, connecting to workspaces, running agents as background daemons. More infrastructural than user-facing.

04Where the meaningful lines are drawn

After looking at all twenty-one products, the meaningful axes aren't “feature X vs feature Y” — they are three structural decisions each product has already made:

Line 1: who runs the agent loop?

  • The user's existing CLI — Claude Squad, CCManager, agent-deck, Conductor, vibe-kanban, brobertsaz/claude-os, claude-mem, Anthropic's own agent view. None of these bundle a chat with their own model. They wrap the user's existing agent.
  • The product itself, embedded — Oyster (today, with bundled OpenCode), OpenClaw OS (defers to OpenClaw), Open WebUI / LobeChat / AnythingLLM / LibreChat (bring-your-own model, but the chat UI is theirs).

The category is migrating to “don't bundle an agent.” The user has Claude Code / Cursor / Codex installed already; they don't need a competing chat surface. This is the single strongest argument for the §7 surgery (drop the chat bar + OpenCode).

Line 2: terminal or browser?

  • Terminal-first — Claude Squad, CCManager, agent-deck, Conductor, Claude agents, Aider, the various dashboards.
  • Browser-first — Oyster, vibe-kanban (cloud + local), Open WebUI, LobeChat, AnythingLLM, LibreChat, OpenClaw OS, brobertsaz/claude-os (with frontend), OpenFang.

This is the cleanest audience split. Terminal-native devs use Claude Squad. Browser-native devs use AnythingLLM / Open WebUI. Oyster is browser-first and that's fine; the question is what's distinctive about Oyster vs the 200k+ stars of self-hosted chat UIs. Answer: spaces, watching external agents, cross-device, publish. None of those are in any browser-first competitor.

Line 3: single agent or many?

  • Claude-Code-only — Anthropic agent view, brobertsaz/claude-os, Conductor, Claudia, claude-control, claude-dashboard variants.
  • OpenClaw-only — OpenClaw OS.
  • Many agents — Claude Squad (Claude/Codex/Gemini/Aider), CCManager (8 agents), agent-deck (Claude/Gemini/OpenCode/Codex), claude-mem (multi-agent claimed), OpenFang (multi-provider routing).

This is the line where Oyster currently sits in the wrong column. Oyster's tagline says “every agent” but ships only Claude Code watching today. CCManager, Claude Squad, and agent-deck have the multi-agent claim shipping in a terminal already. 0.9.0's Cursor/Codex/OpenCode watchers (issue #298) are now load-bearing for Oyster's tagline, not nice-to-have.

The three lines together describe Oyster's lane: “don't run the agent, browser surface, every agent.” Plus the three columns Oyster owns alone in the matrix (spaces, cross-device, publish). That's six structural commitments — a sharp, defensible position. If Oyster ships them and stops doing things that contradict them.

05Oyster's position in the category

Three things are true at once, and they all need to be held in the same sentence:

What Oyster already does that nobody else does

  • Cross-device sync of agent state. Spaces (shipped 0.7.1), memory (shipped 0.8.0), sessions (shipping 0.8.1-beta). No other product in the matrix has this. Vibe Kanban had cloud; that's gone. mem0 / Letta / Cognee have cloud SaaS but it's library infrastructure, not user-facing sync.
  • Publish & share artefacts as URLs with three access modes. share.oyster.to/p/<token>, origin-isolated, password / open / sign-in-required. Nobody else has this primitive at the consumer level.
  • Spaces (projects with attached repos and attributed sessions). AnythingLLM has “workspaces,” but they're chat folders. Oyster's Spaces are project nodes with linked folders, attributed sessions, repo-aware scans. Closest analogue is none of the chat platforms; closer to Linear or GitHub's project sense.
  • MCP-driven workspace where any external agent can read/write the surface. 29 MCP tools. LibreChat consumes MCP servers; Oyster is an MCP server.

What Oyster does that someone else also does

  • Watching Claude Code sessions — now also done by Anthropic claude agents (native, terminal), Claude Squad (multi-agent, terminal), CCManager (multi-agent, terminal). Browser-only is Oyster's differentiator here, plus the integration with spaces/memory/artefacts.
  • Memory across sessions — done by claude-mem (75k), brobertsaz/claude-os, mem0, Letta, Zep, Cognee. Oyster's is FTS5-backed with traceable provenance (R6); cross-device shipped 0.8.0.
  • Markdown artefacts — done by OpenClaw OS, LibreChat's artifacts, basically every chat platform now.
  • Free / Pro tier model — most products are pure OSS with no monetisation; Vibe Kanban tried, failed. Oyster is one of the few with a clear paywall (sync + publish caps).

What Oyster does that probably needs to stop

  • Bundles its own AI engine (OpenCode subprocess). Nobody else in the multi-session category does this anymore. The category has moved to “wrap the user's CLI.” See §7.
  • Lists 7 named artefact kinds (notes, app, deck, wireframe, table, map, diagram). Signals “we generate slides / wireframes / maps” — competing with Keynote / Figma / Mapbox by accident.
  • Has “OS” in the brand — when brobertsaz/claude-os (287★) and OpenFang (17.5k★) both claim the term, and neither means the same thing.

07The yellow pens, re-audited against the category

A separate internal audit (the “yellow pen report,” worktree report/yellow-pen-audit, 2026-05-14) identified seven Sprint-2 code paths still shipping inside the Sprint-5 box. That audit was code-grounded but internally focused — every pen was an inconsistency between Oyster's stated product and the bits on disk. The category map in §02 adds a second lens: how Oyster reads to a user who has the rest of the category in their head. Each of the original seven sharpens. Five new pens emerge that weren't visible from inside the repo.

7.2 — Five new yellow pens, only visible from outside

8. Home-grown memory implementation, when the category has converged on mem0 / Letta / Graphiti / Cognee.

server/src/memory-store.ts is FTS5-backed memory that reimplements what mem0 (55,658★), Letta (22,704★), Graphiti (26,042★), and Cognee (17,218★) have already shipped — with provider-agnostic APIs, MCP integrations, and SaaS clouds. Oyster's defensible claim is memory inside a workspace, not “best memory implementation.” Every engineering hour on memory R&D is one fewer hour on Cursor watchers, cross-device sessions, or publish polish. The fix isn't to rip out memory-store — it's to wrap one of the four under the hood for R2 semantic recall (already on the 0.9.0 roadmap as #321) and stop building competitive home-grown embeddings.

9. “Spaces” terminology collides with AnythingLLM (60,016 stars).

Sixty thousand AnythingLLM users have been trained that “workspaces” means chat folders with documents. Oyster's Spaces mean something different — project nodes with attached repos, attributed sessions, repo-aware scans. A new user importing the AnythingLLM mental model into Oyster gets the wrong picture. The fix isn't to rename — “Spaces” is fine — it's one sentence of marketing copy on the pricing page and README clarifying that Oyster's Spaces are projects, not chat folders.

10. “Bring any AI” on the pricing page, when only Claude Code is watched.

The marketing claim is multi-agent. The code is single-agent. The category benchmarks — CCManager (8 agents shipping), Claude Squad (4+ shipping), agent-deck (4+ shipping) — all already deliver what Oyster's pricing copy promises. This makes the copy aspirational in a way that loses credibility on first user comparison. Issue #298 (Cursor / Codex / Cline watchers) isn't a roadmap nice-to-have; it's the line between honest and aspirational marketing.

11. “Watches your Claude Code sessions” as headline copy, when Anthropic ships it natively.

Not a code pen — a marketing pen. Every screenshot, install screen, and blog post leading with Claude-Code-watching is implicitly competing with Anthropic's own claude agents. They have more resources, deeper integration, and own the platform. Demote Claude-Code-watching from a headline feature to a sub-feature. Lead with the things Anthropic structurally cannot ship: cross-agent (every CLI), cross-device (browser-native, multi-machine), publish-and-share, memory-integrated. Claude-Code-watching is a feature, not the pitch.

12. “OS” in the brand, when OpenFang (17,508★) and brobertsaz/claude-os (287★) own the term in adjacent categories.

Three products with “OS” in the same neighbourhood, none meaning the same thing. The longer Oyster keeps the suffix, the more averaging-out happens in search engines and writers' minds. Originally a Sutherland-style name-level pen; category context upgrades it to an SEO + brand-confusion problem — see the §06 threat table and §08 step 11.

7.3 — The compound effect, sharpened

The original audit said: “Together the seven yellow pens describe a coherent alternate product — Oyster is an AI-driven app workshop.” The category map adds the second half of that sentence:

…and that alternate product is competing in the wrong category, against larger players, with weaker differentiation.
  • Sprint-2 Oyster (AI app workshop) competes with vibeyard, Vibe Kanban, Cline (61.7k★), Aider (44.8k★), Cursor in the “AI builds your app” space. Oyster loses — those products are orders of magnitude bigger and more focused.
  • Sprint-5 Oyster (the shared brain above your agents) competes with nobody for the full umbrella claim. Every competitor owns one slice; the umbrella is empty.
The Sprint-2 yellow pens aren't just legacy code — they're an active drag, pulling Oyster into a category it can't win, away from a category that's empty.

7.4 — The keystone, with category lens

Move #1 (cut the chat bar + bundled OpenCode) was already load-bearing internally. The category context makes it category-defining. One cable, four positionings clarified at once:

  • Removes Oyster from the “self-hosted chat UI” category (311k stars Oyster will never out-build).
  • Removes Oyster from the “AI generates your apps” category (the entire Sprint-2 alternate-product framing collapses).
  • Lands Oyster cleanly in “workspace for your existing agents” — where the umbrella is empty.
  • Resolves the “Bring any AI” pricing-copy contradiction at the source.

This is why §08 step 1 is non-negotiable. Every subsequent step gets easier the moment this one lands.

7.5 — The Morita test, re-applied

If the Sony Walkman will play but will not record, everybody can understand what it's for.

Translating to category-clarity terms:

If Oyster organises but does not chat, everybody can understand what it's for.

The chat bar isn't just a yellow pen. It is the symbol of the Sprint-2 category Oyster is trying to leave. The moment it's gone, the brand promise becomes legible on first install: here is a workspace that watches my agents, remembers their work, and syncs across my machines. It does not run them. I run them.

That is a sentence the category map says nobody else is offering today. It is the umbrella position, articulated in fifteen words. Every yellow pen still in the box is one more word the user has to ignore before they can hear it.

08The order of operations

Same recommendation as v4, with the category context sharpening the priorities. Surgery first, primitives second, rebrand third — all together as the 1.0.0 release.

  1. Cut the chat bar + OpenCode. The keystone. Five of the seven Sprint-2 yellow pens collapse with this one move. The category has moved to “wrap the user's CLI” — Oyster shipping its own agent is now an outlier.
  2. Replace it with a command bar. Keep /s, /o, #, Cmd+K, /p, /u. Route natural-language requests to the user's connected MCP agent (Claude Code, Cursor). First-run is a wizard, not a conversation.
  3. Delete pty-manager.ts, the Ultra Hardcore modal, icon-generator.ts, handleFixError, process-manager.ts.
  4. Delete builtins/zombie-horde/. Re-audit other built-ins against R1–R7.
  5. Collapse 7 artefact kinds to 3 — notes, app, diagram. Absorb the rest into notes. The agent can emit any HTML it needs.
  6. Ship Cursor / Codex / OpenCode watchers (issue #298). Promoted This is no longer 0.9.0-nice-to-have. It's load-bearing for the “every agent” tagline. CCManager and Claude Squad have it shipping today in a terminal; Oyster's window for being the browser-native multi-agent watcher is open and won't stay open forever.
  7. Evaluate embedding mem0 / Letta / Cognee for R2 semantic recall instead of rolling home-grown. New The memory-infrastructure category is converging; integration cost is low, R&D cost of building competitive embeddings is high.
  8. Archive the Sprint-2 sections of docs/plans/archived/oyster-os-design.md.
  9. Pin one tagline everywhere. “Every agent, one shared brain. Sync · Memory · Publish.”
  10. Adopt openui-lang (or an equivalent) as the next-generation app artefact renderer. Lands into a clean foundation.
  11. Rebrand. Drop “OS” from public copy. Lead with curl -fsSL oyster.to/install | sh. GitHub org oyster.to / URL slug oyster-to. Scope npm packages under @oyster-to/*. Urgent The “OS” SEO race has multiple competing products already; staying named “Oyster OS” is a bet that you can outrun OpenFang (17.5k★) and brobertsaz/claude-os (287★) in that namespace. You can't.
1.0.0 is still the right home for all of it. Surgery, multi-agent ingestion, memory embedding, primitive adoption, and rebrand bundled as one “we finished the pivot” release. The cleanest possible product, shipping into a noisy category with a clear position. The roadmap discipline is already in place — every milestone is justified against R1–R7. The codebase just needs to be told it's allowed to lose the limbs that no longer hang off the spine.

09Bottom line

The workspace-for-agents category in May 2026 has roughly twenty serious players. Most own one slice. Three of them — Anthropic claude agents, Claude Squad, CCManager — own multi-session management. Five of them — claude-mem, brobertsaz/claude-os, mem0, Letta, Cognee — own memory. Four — Open WebUI, LobeChat, AnythingLLM, LibreChat — own self-hosted chat surfaces. Two — OpenFang, OpenAgents — claim to be “Agent OS”es of varying scopes. One — OpenClaw OS — owns generative-UI rendering for one ecosystem.

Oyster's claim is the union. Sit above all the agents, remember everything they did, organise by project, sync across devices, publish any of it. No competitor today holds three of those columns together; nobody holds four. That union is where the unique value lives and where the defensible moat is.

The hardest truth: the union doesn't ship for free. Oyster has to be best-in-class at five things that other products are individually focused on. Better than CCManager at multi-agent watching, better than claude-mem at cross-session memory, better than AnythingLLM at workspaces, better than LibreChat at artifacts-as-URLs, better than no-one-else at cross-device sync. That's the cost of the umbrella.

The compensating truth: nobody else is even trying. The umbrella is empty. Twenty competitors below, each owning their slice, none reaching upward. Oyster's defensible move is to commit to the umbrella visibly — drop the bundled agent, drop the OS suffix, ship the multi-agent watchers, integrate the memory infrastructure, keep shipping cross-device sync — and let each single-purpose product be very good at their slice while Oyster is the only thing tying them all together.

If forced to commit to one sentence: the umbrella is the moat. Build the umbrella, not another slice.
